# Distill v3.1 — Experience & Lessons Learned

> 内部文档，记录真实的开发过程、踩坑细节和决策依据。不做对外发布筛选——这份文档的目的是给未来的自己（或协作者）留下完整上下文，而不是公关材料。挑选哪些内容适合写进 devpost 是另一个独立的步骤。

## 1. B站CDN静默截断：本次开发最有价值的debug故事

### 现象

用 yt-dlp 下载B站视频音频，命令执行成功、退出码0、文件确实生成、文件大小看起来合理（4-5MB），但用 `ffprobe` 检查实际播放时长，发现只有 **5秒左右**，无论视频原本是几分钟还是几十分钟。

这不是一个会报错的失败——这是最难处理的那种bug：**看起来一切正常，但数据是错的**。

### 排查过程（按实际发生顺序）

1. 一开始怀疑是 cookie 不完整（只导出了 SESSDATA/bili_jct/DedeUserID 三个字段），重新用浏览器插件导出了包含 buvid3/buvid4/b_nut 等全部字段的完整 cookie。问题依旧。
2. 怀疑是本地代理（开发机配了 `http://127.0.0.1:7897`）截断了流量，加 `--proxy ""` 强制直连。问题依旧。
3. 怀疑是 yt-dlp 的章节提取（`Extracting chapters`）把视频错误分段，只下载了第一个5秒的章节。加 `--no-split-chapters`。问题依旧。
4. 用 `ffprobe` 详细检查文件元数据，发现 `bitrate: 68kb/s` 远低于声明的96kbps，怀疑文件本身损坏而非真的只有5秒数据。
5. **关键转折**：手动测试时偶然加了 `-k`（保留中间文件）参数，这次 yt-dlp 自动选择了 `视频流+音频流` 一起下载再合并提取（格式 `100026+30280`），而不是之前一直选择的纯音频流（`30280`）。这次下载到的是完整的430秒音频。
6. 去掉 `-k` 重新测试纯音频流下载——这次又成功了，时长正常。

### 真正的根因（推测，未100%验证）

最可能的解释：B站对纯音频流（`30280`格式）使用的CDN节点（`mcdn.bilivideo.cn`，这是用户共享带宽/P2P-CDN节点）存在**偶发性的不稳定**，有时会提前关闭连接或返回不完整的响应体，但 HTTP 状态码仍然是200，yt-dlp 没有发现异常就把不完整的数据当作下载成功处理了。这不是一个100%复现的bug，是个概率性的CDN质量问题。

### 最终解决方案

不去"修复"yt-dlp或猜测哪个参数组合能稳定复现，而是接受这是个**概率性的环境问题**，用工程手段兜底：

1. 下载完成后立即用 `ffprobe` 校验音频时长，设定一个最低阈值（10秒）作为"明显异常"的判定标准
2. 如果时长异常，自动重试，重试时**强制改变下载策略**（第一次用纯音频流，第二次起切换到"视频+音频再提取"），利用两条不同的下载/CDN路径增加成功概率
3. 每次重试都加 `--force-overwrites --no-continue`，因为yt-dlp默认会看到"文件已存在"就跳过重新下载，导致重试逻辑形同虚设——**这是另一个独立的坑**，差点让前面的重试机制完全失效

### 这里的教训

- **"成功执行 + 文件存在" 不等于 "数据正确"**。任何涉及外部数据获取的pipeline，都应该有一个独立于"进程是否报错"的数据有效性校验步骤（这里是用 ffprobe 测时长，其他场景可能是校验文件hash、内容长度、关键字段是否存在等）。
- **重试逻辑必须确认真的在"重试"**，而不是命中缓存返回同一个坏结果。这个问题不会在第一次测试时暴露，因为第一次必然是真实执行；只有在系统设计了重试机制之后，才可能因为底层工具的缓存行为让重试变成空转——这是一类容易被忽略的"重试的重试不够重试"问题。
- 遇到概率性、难以稳定复现的环境问题时，"换一种完全不同的实现路径来重试"往往比"反复调试同一条路径为什么失败"更高效。两条命令在我手动测试时表现不一致（一次成功一次失败，参数几乎相同），说明问题出在环境侧的随机性，而不是命令本身的逻辑错误——这种情况下继续深挖命令参数是低收益的。

## 2. Windows 平台的 cookie 提取困境

### 问题链条

为了让 yt-dlp 访问B站需要登录才能看到的内容（实测发现：连未登录可见的普通视频，在某些请求模式下也会返回412），需要给 yt-dlp 提供 B站登录态。yt-dlp 内置了 `--cookies-from-browser` 功能，理论上可以直接读取浏览器（Chrome/Edge）已保存的cookie，不需要手动导出。

但在 Windows 上，这个功能从某个 Chromium 版本之后被**官方设计为永久失效**——这不是bug，是 Chromium 团队主动的安全加固：cookie使用DPAPI加密存储，绑定到生成它的具体进程会话，外部工具无法再批量解密，这是为了堵住"任何脚本都能窃取浏览器cookie"的安全漏洞（参见 yt-dlp issue #10927，以及对应的 Chromium 安全公告）。

### 这意味着什么

任何依赖"自动从浏览器读取cookie"的自动化工具，在 Windows + Chromium内核浏览器（Chrome/Edge）上都会遇到同样的死路。**这不是配置问题，没有命令行参数可以绕过**，唯一可行路径是用浏览器插件手动导出cookie到文件，再用 `--cookies file.txt` 读取。

### 额外的格式坑

浏览器插件导出的cookie文件用 Netscape格式（tab分隔的文本），但实际操作中遇到两个格式问题：

1. 插件导出的文本在某些查看器/编辑器中会被自动渲染成 Markdown 链接格式（`.www.bilibili.com` 变成 `.[www.bilibili.com](https://www.bilibili.com)`），如果直接复制粘贴这种渲染后的文本会导致解析失败。需要确保拿到的是纯文本，不是渲染后的显示效果。
2. Netscape格式对"域名是否以点号开头"和"第六列(TRUE/FALSE)"有强制关联校验——以点号开头的域名（如 `.bilibili.com`，表示对所有子域名生效）第六列必须是 `TRUE`，否则 Python 的 `http.cookiejar` 解析器会直接抛 `AssertionError` 拒绝加载整个文件。手动修复用了一个简单的正则替换脚本批量纠正。

## 3. 一个差点酿成真实安全事故的细节

在排查cookie问题的过程中，为了让我能复现并调试问题，对方在对话中直接粘贴了完整的 `SESSDATA`、`bili_jct`、`DedeUserID` 三个值——这三个字段组合起来等同于完整的B站账号登录凭证，任何拿到这三个值的人都可以直接操作该账号，无需密码、无需二次验证。

这是一个**应该被严肃对待的安全教训**，不只是"这次没出事就算了"：

- 任何形式的"把凭证发给AI助手调试"都应该被默认视为凭证已泄露，事后必须撤销（退出所有设备登录，使旧的session token失效）。
- 更好的做法是从一开始就避免这种情况：调试cookie问题时，应该让用户在本地自行验证（运行我提供的命令，把*结果*而不是*凭证本身*回传），或者用占位符/打码的方式描述问题。这次没有在最早期就采用这种方式，是过程设计上的疏漏。
- 如果未来这个项目要做成可以给团队成员协作开发的程度，"如何安全地调试需要敏感凭证的功能"应该形成一个文档化的规范，而不是临场处理。

## 4. 架构决策：为什么LLM Router现在是内嵌的，不是独立服务

最初的项目规划（README、architecture.md的早期版本）里，LLM Router被描述为 `services/llm-router/` 下的独立服务，有自己的 `package.json`、可以 `pnpm dev` 单独启动。

实际开发时发现：当前唯一会调用这个router的消费者就是 Next.js 前端自己。在只有一个消费者的情况下，把它拆成独立进程意味着：

- 需要解决跨包的TypeScript类型共享或者改用HTTP调用（额外的网络开销和序列化成本）
- 需要额外的进程管理（开发时要同时启动两个`pnpm dev`，部署时要多维护一个服务的健康检查/重启策略）
- 在Next.js的API route里，`lib/`下的TS代码可以被直接import，由Next.js自带的编译器（Turbopack）一起编译，没有任何额外配置

所以最终选择把router的运行时代码复制进 `apps/web/lib/`，`services/llm-router/`保留作为这套逻辑的"源头"和独立测试场地（可以用`tsx`直接跑测试脚本，不需要启动整个Next.js）。

这是一个**刻意的、有理由的简化**，不是图省事走捷径——但需要清楚记录下来，因为：

1. 如果未来有第二个消费者（比如一个独立的批处理脚本，或者要支持的n8n工作流），这个决策需要重新评估
2. 当前计划是把router的编排逻辑迁移到 n8n，这意味着"内嵌在Next.js里"只是一个过渡阶段的状态，不是终态

## 5. 一些更轻量的小教训（不构成独立章节，但值得记一笔）

- **pnpm的`approve-builds`机制**：新版本pnpm默认拦截依赖的postinstall脚本执行（出于供应链安全考虑），这在第一次`pnpm install`时经常让人以为是依赖装失败了，实际上是需要手动跑`pnpm approve-builds`确认。第一次遇到容易浪费时间排查"为什么装不上"。
- **跨平台命令行语法的反复踩坑**：同一个任务在 PowerShell / CMD / Git Bash 下，反斜杠续行符、多行heredoc、环境变量加载方式（`--env-file`）都不一样，开发过程中反复在三者之间切换导致了多次"命令无法识别"的报错。如果团队/未来的自己主要在Windows开发，提前固定一种shell（建议Git Bash，因为大部分命令片段是Unix语法）会减少很多摩擦。
- **中文字符不能直接放进HTTP Header**：早期实现里环境变量或某个意外路径把中文字符传进了`fetch`的`headers`对象，触发了`ByteString`编码错误。这类问题报错信息相对隐晦（"character at index N has a value greater than 255"），需要知道这是Header编码限制才能快速定位。

## 写在最后

这次开发暴露的核心主题是：**当你处理别人的基础设施（B站CDN、第三方ASR API、Windows系统层的cookie加密机制）时，大部分bug的根因不在你自己的代码里，而在于这些外部系统的行为假设和你的预期不一致**。无论是CDN偶发返回截断数据、yt-dlp默认行为掩盖了重试失效、还是Windows安全机制让一个曾经能用的功能突然失效——这些都不是"写错代码"导致的，而是"对外部系统的行为模型理解不完整"导致的。这类问题的应对方式通常不是更仔细地写代码，而是**为不确定性设计防御机制**（数据校验、强制重试路径切换、明确的失败阈值），这和纯算法/业务逻辑bug的调试思路是不同的肌肉。
