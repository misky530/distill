# Distill v3.1 — Engineering Notes

> 技术复盘文档，记录开发过程中几个有价值的debug故事和架构决策依据。

## 1. B站CDN静默截断：一次值得记录的debug故事

### 现象

用 yt-dlp 下载B站视频音频，命令执行成功、退出码0、文件确实生成、文件大小看起来合理（4-5MB），但用 `ffprobe` 检查实际播放时长，发现只有 **5秒左右**，无论视频原本是几分钟还是几十分钟。

这不是一个会报错的失败——这是最难处理的那种bug：**看起来一切正常，但数据是错的**。

### 排查过程

1. 一开始怀疑是 cookie 不完整，重新导出了包含全部字段的完整 cookie。问题依旧。
2. 怀疑是本地代理截断了流量，加 `--proxy ""` 强制直连。问题依旧。
3. 怀疑是 yt-dlp 的章节提取把视频错误分段，只下载了第一个片段。加 `--no-split-chapters`。问题依旧。
4. 用 `ffprobe` 详细检查文件元数据，发现实际码率远低于声明值，怀疑文件本身损坏而非真的只有5秒数据。
5. **关键转折**：手动测试时偶然加了 `-k`（保留中间文件）参数，这次 yt-dlp 自动选择了"视频流+音频流"一起下载再合并提取，而不是之前一直选择的纯音频流。这次下载到的是完整时长的音频。
6. 去掉 `-k` 重新测试纯音频流下载——这次又成功了，时长正常。

### 根因（推测，未100%验证）

最可能的解释：B站对纯音频流使用的某些CDN节点（用户共享带宽/P2P-CDN性质）存在**偶发性的不稳定**，有时会提前关闭连接或返回不完整的响应体，但 HTTP 状态码仍然是200，yt-dlp 没有发现异常就把不完整的数据当作下载成功处理了。这是个概率性的CDN质量问题，不是100%复现的bug。

### 解决方案

不去"修复"yt-dlp或猜测哪个参数组合能稳定复现，而是接受这是个**概率性的环境问题**，用工程手段兜底：

1. 下载完成后立即用 `ffprobe` 校验音频时长，设定最低阈值（10秒）作为"明显异常"的判定标准
2. 如果时长异常，自动重试，重试时**强制改变下载策略**（第一次用纯音频流，第二次起切换到"视频+音频再提取"），利用两条不同的下载/CDN路径增加成功概率
3. 每次重试都加 `--force-overwrites --no-continue`，因为yt-dlp默认会看到"文件已存在"就跳过重新下载——这会让重试逻辑形同虚设，是另一个独立的坑

### 教训

- **"成功执行 + 文件存在" 不等于 "数据正确"**。任何涉及外部数据获取的pipeline，都应该有一个独立于"进程是否报错"的数据有效性校验步骤。
- **重试逻辑必须确认真的在"重试"**，而不是命中缓存返回同一个坏结果。
- 遇到概率性、难以稳定复现的环境问题时，"换一种完全不同的实现路径来重试"往往比"反复调试同一条路径为什么失败"更高效。

## 2. Windows 平台的 cookie 提取限制

为了让 yt-dlp 访问B站需要登录才能看到的内容，需要给 yt-dlp 提供B站登录态。yt-dlp 内置了 `--cookies-from-browser` 功能，理论上可以直接读取浏览器已保存的cookie。

但在 Windows 上，这个功能从某个 Chromium 版本之后被**官方设计为永久失效**：cookie使用DPAPI加密存储，绑定到生成它的具体进程会话，外部工具无法再批量解密（参见 yt-dlp issue #10927）。这不是配置问题，没有命令行参数可以绕过，唯一可行路径是用浏览器插件手动导出cookie到文件，再用 `--cookies file.txt` 读取。

实操中还遇到两个格式坑：
1. 浏览器插件导出的Netscape格式cookie文件，在某些查看器/编辑器中会被自动渲染成 Markdown 链接格式，直接复制粘贴渲染后的文本会导致解析失败，需要确保拿到的是纯文本。
2. Netscape格式对"域名是否以点号开头"和"第六列(TRUE/FALSE)"有强制关联校验，不满足会让 Python 的 `http.cookiejar` 解析器直接抛 `AssertionError` 拒绝加载整个文件。

## 3. 架构决策：为什么LLM Router现在是内嵌的，不是独立服务

最初的项目规划里，LLM Router被设计为独立服务，可以单独启动、单独部署。实际开发时发现：当前唯一会调用这个router的消费者就是 Next.js 前端自己。在只有一个消费者的情况下，拆成独立进程意味着额外的类型共享/网络调用成本，以及额外的进程管理负担，而收益几乎没有。

所以最终选择把router的运行时代码内嵌到 Next.js 的 `apps/web/lib/` 中，由 Next.js 自带的编译器一起编译，没有任何额外配置。独立服务的源码作为这套逻辑的"源头"保留，便于独立测试。

这是一个**刻意的、有理由的简化**：

1. 如果未来有第二个消费者（比如一个独立的批处理脚本，或者要支持的n8n工作流），这个决策需要重新评估
2. 当前计划是把router的编排逻辑迁移到 n8n，这意味着"内嵌在Next.js里"只是一个过渡阶段的状态，不是终态

## 4. 一些更轻量的小教训

- **pnpm的`approve-builds`机制**：新版本pnpm默认拦截依赖的postinstall脚本执行（出于供应链安全考虑），第一次`pnpm install`时容易让人误以为依赖装失败，实际上需要手动跑`pnpm approve-builds`确认。
- **跨平台命令行语法**：同一个任务在 PowerShell / CMD / Git Bash 下，反斜杠续行符、多行heredoc、环境变量加载方式都不一样，提前固定一种shell能减少很多摩擦。
- **中文字符不能直接放进HTTP Header**：环境变量或某个意外路径把中文字符传进了`fetch`的`headers`对象，会触发`ByteString`编码错误，报错信息相对隐晦，需要知道这是Header编码限制才能快速定位。

## 写在最后

这次开发暴露的核心主题是：**当你处理别人的基础设施（B站CDN、第三方ASR API、Windows系统层的cookie加密机制）时，大部分bug的根因不在自己的代码里，而在于这些外部系统的行为假设和预期不一致**。这类问题的应对方式通常不是更仔细地写代码，而是**为不确定性设计防御机制**（数据校验、强制重试路径切换、明确的失败阈值），这和纯算法/业务逻辑bug的调试思路是不同的肌肉。
